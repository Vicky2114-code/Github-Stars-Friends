# Design: Repo Deep-Dive (formerly "GitHub Stars & Friends")

## Problem Statement

The original spec described "GitHub + LinkedIn + Product Hunt + Discord for developers." Through interrogation it collapsed to a sharper problem: **repo owners — including the user — currently rely on GitHub's native UI and ad-hoc tools (star-history.com) to understand how their projects are growing, and the existing visualizations are basic.** The wedge is a beautiful, shareable, no-login page per repo that answers "how is this project doing, and is it AI-discoverable?"

## Demand Evidence

User self-reported behavior, verbatim:
> "I track my own GitHub stats manually."

This is the only validated demand signal at this stage — the user IS the user. No external interviews, no waitlist, no paying customers. The user explicitly acknowledged: *"Honestly, none yet — this is my idea."* The strength is that the user has real weekly pain; the weakness is sample size of one. Distribution validates the rest.

Adjacent market evidence (not validation, but proof the category exists):
- `star-history.com` — high-traffic, basic visualization
- `gitstar-ranking.com` — leaderboard, basic UI
- `repostats.dev`, `ossinsight.io` — newer entrants, varying polish

## Status Quo

The user currently opens GitHub's native repo Insights tab and star-history.com, manually, repeatedly. The pain isn't *missing data* — GitHub exposes it all via API and partial UI. The pain is **synthesis and aesthetics**: trends, comparisons, contributor health, and "is this README good enough" are scattered or absent.

## Target User & Narrowest Wedge

**Target user:** The repo owner curious about their own project's trajectory. Secondary: developers evaluating libraries they might depend on.

**Wedge:** A single page at `repo-dive.app/<owner>/<repo>` (no login required to view) that renders within ~2 seconds and contains:
1. Star/fork/contributor growth charts reconstructed from the GitHub `stargazers` API (each star has `starred_at`).
2. A one-paragraph AI summary: "What is this repo, who would benefit, what's missing."
3. **The differentiator** — an AI Discoverability check: ask GPT-4o/Claude/Gemini "what libraries would you recommend for [topic from repo]?" and report whether the repo surfaces. This is the sharper version of the spec's vague "AI Reach Score" — a concrete question developers genuinely worry about and no other tool answers.
4. A "Made with repo-dive" share footer for the viral loop.

Everything else from the original spec (friends, leaderboards, achievements, activity feed, notifications, email digests, owner accounts, comparison view) is **explicitly cut from MVP** and parked in v2+.

## Constraints

- **Ship in ≤5 days.** Velocity is the primary constraint.
- **Solo developer.** No team to coordinate.
- **Free tier only at launch.** No payment infrastructure in MVP.
- **GitHub rate limits**: 60 req/hr anonymous, 5000/hr with PAT. Mitigated by edge caching.
- **OpenAI cost**: ~$0.01-0.05 per AI score (gpt-4o-mini, cached 24h). Budget: assume $10/mo at launch traffic.

## Premises (agreed)

1. The real pain is repo owners wanting better growth insight than GitHub's native UI provides.
2. No-login viewer is the unlock — paste-a-URL beats authenticated dashboards at this stage because it sidesteps cold-start.
3. AI Reach Score earns its place only as a concrete question: "Will Claude/GPT/Cursor recommend this repo when asked for libraries in this category?" — not a generic popularity number.
4. Friends, leaderboards, social feed, achievement badges, notifications — all cut from MVP. They are v2+ features that only make sense after the deep-dive page has weekly users.

## Approaches Considered

### Approach A: Frontend-only MVP (CHOSEN)
Next.js 15 (App Router) on Vercel. No FastAPI, no PostgreSQL, no Redis. All GitHub data fetched on demand via the GitHub REST/GraphQL API. AI scoring is a single OpenAI call cached in Vercel KV (24h TTL). Charts via Recharts. ShadCN UI + Tailwind. Auth deferred — only needed if/when "claim this repo" lands.
- Effort: S (3-5 days)
- Risk: Low
- Reuses: Vercel platform, GitHub API, OpenAI API, ShadCN component library

### Approach B: Full Stack from Day 1
Next.js frontend + FastAPI backend + PostgreSQL + Redis + Celery worker + SendGrid. All spec features in v1. Owner accounts, claimed repos, email digests, leaderboards, comparison.
- Effort: L (2-3 weeks)
- Risk: Med — common failure mode is "build everything, ship nothing"
- Cut: deferred to Phase 2

### Approach C: CLI-first
`npx repo-dive owner/repo` generates a self-contained HTML report. No hosting. Distribution via npm.
- Effort: S (5 days)
- Risk: Low
- Cut: web URL sharing is more frictionless than CLI install. Can be added later as a power-user feature.

## Recommended Approach

**Approach A.** Lowest path-to-shipped product, lowest debug surface, full upgrade path to B without throwing code away. The Next.js codebase can grow a FastAPI sidecar later for the features that actually require persistent state (email digests, claimed repos, leaderboard snapshots). The MVP itself is fully stateless.

## Open Questions

1. **GitHub PAT model for viewer requests.** Anonymous rate limit (60/hr) is dangerous if a single page goes viral on HN. Options: (a) use a server-side PAT for all API calls — risk of single-account ban; (b) require viewers to sign in with GitHub for "premium" features — adds friction; (c) GitHub App with installation-scoped tokens — most correct, more setup. **Decision needed before coding.**
2. **AI Discoverability sample size.** Asking GPT-4o once is non-deterministic — a repo might surface in 3/10 runs. Should the score be an average over 5 runs? Cost vs. signal tradeoff.
3. **Repo URL parsing.** `/<owner>/<repo>` route is clean but breaks on transferred/renamed repos. Add a redirect step or accept stale links?
4. **What "shareable URL" actually looks like.** OpenGraph image generation for nice Twitter/LinkedIn cards is its own ~half-day of work — worth it for v1?

## Success Criteria

Concrete and measurable, in priority order:
1. **Week 1:** The user (you) opens `repo-dive.app/<your-username>/<your-repo>` instead of GitHub's Insights tab. Replaces the manual workaround.
2. **Week 2:** Share the URL of one of your own repos on Twitter/HN. Measure: does anyone click? Does anyone paste their own repo?
3. **Week 4:** ≥50 unique repos viewed per week (cheap to measure with PostHog/Plausible). If yes, validate; if no, the wedge is wrong or distribution is wrong.
4. **Month 2:** Decision gate — promote to Phase 2 (auth + accounts + persistence) only if Week 4 metric hits.

## Distribution Plan

- **Hosting:** Vercel (free tier covers v1).
- **Domain:** `repo-dive.app` (assumption — verify available; fallback `repo-deep-dive.com`).
- **Launch surface:** Twitter (paste a screenshot of your own repo's page), HN Show HN, Indie Hackers, Reddit r/github.
- **No CI/CD beyond Vercel's git integration in v1.**
- **No paid acquisition.** Word-of-mouth via the share footer.

## Dependencies

- GitHub API (REST + GraphQL) — must understand rate limits before coding.
- OpenAI API — gpt-4o-mini for cost. Optional: Anthropic + Google AI Studio for the multi-model discoverability check.
- Vercel KV for the AI-score cache.
- ShadCN UI + Recharts (no surprises).

## The Assignment

**Before writing any code:** Open GitHub's native Insights tab for your two best repos. Open star-history.com for the same two. Take screenshots of each. List, on paper, the **three specific things** GitHub/star-history *don't* show that you most want to see. That list is your v1 feature set. Anything not on the list is YAGNI for week 1.

(This forces the wedge to come from observation, not from the spec. If your list contains "friends" or "leaderboard," good — disagree with this design doc. If it contains "growth velocity vs. similar repos," "is my README sparse?", or "will Cursor recommend this?", great — those are the features.)

## What I noticed about how you think

- **You were honest when honesty was uncomfortable.** When asked for demand evidence you said "honestly, none yet — this is my idea" instead of dressing it up as "I've talked to many developers." That's the rarest founder trait in this room. Most people lie to themselves first.
- **You picked the wedge with the best growth-loop physics** (shareable URL vs. personal dashboard) rather than the one closest to your original spec. You changed your mind under pressure of evidence — that's the trait that separates founders from idea-havers.
- **You accepted "cut from MVP" for features you wrote in the spec.** Friends, leaderboards, badges — all your ideas, all cut. Most people defend their spec; you let it go. The product is better for it.
- **You haven't done customer discovery yet.** The Assignment exists because of this — schlep-blindness is your friend at this stage, but it has a half-life. After week 2 you need ≥3 real conversations with other developers, not just your own behavior, or the wedge ossifies around an N=1 sample.

## Decisions locked from this review

| # | Decision | Choice | Reason |
|---|---|---|---|
| D1 | GitHub rate limit strategy | Server-side PAT in env var + Vercel KV cache (24h) + circuit breaker | Single point of control, viable to ~50k pageviews/day, GitHub App is v1.1 |
| D2 | Stargazers pagination | Sampling (first page + last page + 8 evenly-spaced middle) | 10 requests max per repo. star-history.com pattern. ~95% chart accuracy |
| D3 | AI Discoverability methodology | Gemini Flash × 3 prompts per repo | ~$0.01/repo. 24h cache. Score = "mentioned in N/3 prompts". Variance from prompts, not models |

### Request flow for `/<owner>/<repo>`

```
User hits /vercel/next
   │
   ▼
Next.js Server Component renders
   │
   ├── [parallel] getRepoMeta(owner, repo)         ──► KV? hit: 5ms / miss: GitHub /repos
   ├── [parallel] getStarHistorySamples(...)       ──► KV? hit: 5ms / miss: 10 sampled pages
   ├── [parallel] getContributorStats(owner, repo) ──► KV? hit: 5ms / miss: GitHub /contributors
   ├── [parallel] getReadmeAnalysis(owner, repo)   ──► KV? hit: 5ms / miss: GitHub + small LLM call
   └── [parallel] getAIDiscoverability(owner, repo)──► KV? hit: 5ms / miss: 3 Gemini calls
   │
   ▼
Page renders with charts, score, summary
Edge cache: s-maxage=86400, stale-while-revalidate=43200
```

Cold render budget: ≤3s. Warm cache render: ≤300ms.

**Nothing — green-field project.** No existing code to reuse. The Next.js ecosystem covers everything:

- Image optimization → next/image

## Failure modes

For each lib/ module, one realistic production failure scenario and whether the plan covers it:

| Module | Failure scenario | Test? | Error handling? | User sees? |
|---|---|---|---|---|
| `lib/github.ts` | PAT revoked / expired | ✓ unit | ✓ circuit breaker → cached/degraded | ⚠ "Showing cached data" banner |
| `lib/github.ts` | Repo is private / 404 | ✓ unit | ✓ not-found.tsx | ✓ clear "repo not found" |
| `lib/github.ts` | Rate limit < 500 | ✓ unit | ✓ circuit breaker | ⚠ "Showing cached data" banner |
| `lib/star-history.ts` | Repo has 0 stars | ✓ unit | ✓ empty-state component | ✓ "No stars yet — your first star awaits" |
| `lib/star-history.ts` | API returns stars with `null` `starred_at` | ✓ unit | ✓ filter nulls | (silent — degraded chart accuracy) |
| `lib/ai-discoverability.ts` | Gemini API down | ✓ unit | ✓ try/catch → score: null | ⚠ "AI score unavailable" |
| `lib/ai-discoverability.ts` | Gemini returns garbage / non-JSON | ⚠ **GAP** | ✗ **GAP** | ⚠ **CRITICAL** — would crash page |
| `app/[owner]/[repo]/page.tsx` | One of the 5 parallel calls fails | ✓ e2e | ✓ error boundaries per card | ✓ that card shows degraded state |
| `app/[owner]/[repo]/page.tsx` | Slow render (>3s) | ✗ (perf budget unmeasured) | N/A | User sees blank for >3s |

**Critical gap flagged:** Gemini garbage response handling. Add explicit JSON-schema validation in T4 with fallback to `score: null`.

## Worktree parallelization

**Mostly sequential** — small project with shared `lib/` and `app/` directories.

| Lane | Tasks | Why sequential/parallel |
|---|---|---|
| Lane A | T1 (bootstrap) | Blocks everything |
| Lane B | T2, T3, T4, T5 | Could parallelize across worktrees — each touches only its own lib/ file. Tests too. |
| Lane C | T6, T7 | Sequential — T7 components depend on T6 page scaffolding |
| Lane D | T8, T9, T10, T11 | Parallelizable — each touches different files |
| Lane E | T12, T13 | Sequential — deploy after test passes |

**Realistic parallel speedup:** ~25%. The big serial chunk (T6+T7) is half the work. Not worth setting up worktrees for an MVP this small.

## Failure modes — critical gaps to fix in implementation

1. **Gemini garbage JSON response** in `lib/ai-discoverability.ts` — add zod schema validation, fallback to `score: null`. Without this, one bad LLM response crashes the page.
2. **No analytics from day 1** — T11 added. Without it the Success Criteria ("≥50 unique repos viewed per week") can't be measured.
3. **No retry logic on transient GitHub 5xx** — `lib/github.ts` should retry once with 500ms backoff before tripping the circuit breaker.
